Today I set out to build a portfolio website.

I have attempted this many times and failed to finish. I wanted to build it from scratch using the latest and greatest tools like React, Typescript, and Redux.

I have found that it simply takes too long for a simple project. It’s easy for scope creep to accumulate and to end up feeling too bogged down to complete it.

So what changed? How did I make this website?

In order to finally make this website, I had to take a step back and use a tool that could help me build the website faster. I found Hugo and its many website templates. Next thing I knew I already had a professional looking website that just needed to be filled out.

While it’s nice to know how to use the big modern tools to build custom software, it’s also important to know when to use them. I think for a simple website project like this it makes sense to hold off on the large tools and use something simple and open source that can do the heavy lifting.

Making pragmatic architectural choices is an important lesson in software development.

Us developers want to use the fanciest newest tools, but we should be pragmatic about our choices. How much time do we want to spend on this project? How much time do we have?

In larger projects there are often time restraints. Clients often expect new features by hard deadlines. It becomes the engineers role to determine the best approach. This is where we make this pragmatic trade-off: do we use the fancy tool that has a lot of flexibility or do we choose the tool that can 10x our speed?

It’s important to apply a cost benefit analysis in this way to determine the correct approach to the architectural decisions of software. There are more factors than just time and money to consider as well: extensibility, reliability, scaleability, etc.

I could have chosen a tool like SquareSpace that builds the entire website head to toe, but I still wanted to have that personal “Hey! Look! I’m a Software Developer!” touch. I can still make some developer level choices, integrate the project with Git, and eventually I will host this though github pages.

So today, I chose time as my important deciding factor ( with a side of hands-on developing ). I want this website up and running fast so I can go focus on other projects to build that will take more time and I want to be able to integrate it with Git / Github. This website can be simple, so I chose the fastest option that allowed me developer integration.
